## 1. 종합 결론

Plate Service는 현재 기능 중심의 구조는 비교적 잘 갖춰져 있다. 특히 일반 고객지원, 식당 파트너, 관리자 영역이 라우트와 권한 기준으로 분리되어 있고, 준비되지 않은 기능은 메뉴에서 숨기거나 준비 중 화면으로 명확히 처리하려는 방향이 보인다. 이 점은 사용자에게 과장된 기대를 주지 않는다는 면에서 좋은 친절함이다.

다만 다음 단계에서는 단순히 "기능이 동작하는가"보다 "사용자가 지금 어떤 상태에 있고, 다음에 무엇을 해야 하며, 왜 이 절차가 필요한지"를 더 분명히 알려야 한다. 입점 신청 상태, Q&A 공개 범위, 1:1 비공개 문의 접수는 1차 보강이 되었지만, 서버 기반 처리 이력, 보완 항목별 수정 흐름, 내 문의 목록, 답변 알림, 관리자 알림과 작업 우선순위는 아직 기능으로 남아 있다.

현재 앱은 기본 운영 흐름을 지원하는 수준을 넘어, 일반 사용자와 식당 점주의 진입 맥락을 더 명확히 구분하는 단계까지 개선되었다. 운영 친화적인 서비스로 올라가기 위해서는 서버 처리 이력, 내 문의 목록, 알림, 미디어 관리 고도화가 다음 우선순위다.

## 1.1 반영 현황

### 완료

- FAQ 생성 시 남던 콘솔 로그를 제거했다.
- FAQ 목록 오류 문구를 개발자형 표현에서 사용자형 표현으로 변경했다.
- FAQ 삭제와 매장 삭제를 브라우저 기본 확인창이 아니라 앱 내부 확인 다이얼로그로 변경했다.

- 관리자 상단의 `통합 검색`을 실제 동작 범위에 맞춰 `승인 검색`으로 조정했다.
- 관리자 알림 버튼과 대시보드 `전체 활동 보기`는 실제 기능처럼 보이지 않도록 준비 중 상태로 낮췄다.
- Q&A 작성 영역에 공개 범위, 개인정보 입력 주의, 이메일 사용 목적 안내를 추가했다.
- Q&A 작성 시 로그인 사용자도 입력한 답변 이메일이 요청 payload에 포함되도록 정합성을 맞췄다.
- 공개 Q&A 목록과 질문 등록 화면을 분리했다. 공개 목록은 `/qna`, 등록은 `/qna/new`에서 처리한다.
- 공개 Q&A를 카드형 타임라인에서 게시판형 목록으로 변경하고, 답변은 펼쳐서 확인하도록 정리했다.
- 1:1 비공개 문의 접수 화면을 `/qna/private`로 추가하고 공개 Q&A와 동선을 분리했다.
- 공개 Q&A 목록에서 `isPublic: false` 또는 `hidden` 상태 문의가 노출되지 않도록 프론트 방어 로직을 추가했다.
- 회원가입 화면에 회원 ID/비밀번호 입력 조건 안내와 약관/개인정보 처리방침 전문 링크를 추가했다.
- 입점 신청 상세에 상태별 다음 행동 안내 패널을 추가했다.
- FAQ 목록에 분류와 검색어 필터를 추가했다.
- Q&A 목록에 문의 유형과 처리 상태 필터를 추가했다.
- 입점 신청 상세에 현재 진행 단계를 보여주는 미니 타임라인을 추가했다.
- 매장 관리와 매장 상세에 노출 상태별 의미 안내를 추가했다.
- 고객지원 공개 FAQ를 일반 게시판/아코디언형 목록으로 정리하고 더 보기 동선을 추가했다.
- 로그인 상태의 일반 사용자/식당 점주 진입을 고정된 모드 스위치로 정리했다.
- 고객지원 헤더와 본문 카드의 폭, 배경, 그림자 톤을 맞춰 화면 정합성을 높였다.

### 부분 완료

- 입점 신청 상태 맥락은 상태별 안내 패널과 미니 타임라인까지 반영했다. 다만 서버 기반 전체 처리 이력, 보완 항목별 수정 바로가기, 제출 버전 비교는 아직 남아 있다.

- 상단 모드 전환은 진입 맥락을 단순화했다. 다만 이것은 권한 변경 기능이 아니라 목적지 전환 UI이며, 실제 권한 보호는 기존 라우트 가드에 의존한다.

## 3. 잘 되어 있는 점

### 3.1 사용자 영역 분리

앱은 크게 고객지원, 식당 파트너, 내부 관리자 영역으로 나뉜다.

- 고객지원: FAQ, Q&A, 약관, 개인정보 처리방침
- 식당 파트너: 입점 신청, 신청 현황, 승인된 매장 관리
- 관리자: 대시보드, 승인 관리, FAQ/Q&A 관리, 회원 모니터링

`src/components/AppShell.js`와 `src/admin/components/AdminShell.js`에서 각 영역의 헤더와 내비게이션이 분리되어 있어 사용자가 현재 자신이 어떤 영역에 있는지 파악하기 쉽다. 특히 공개 화면 상단에 `일반 사용자`와 `식당 점주` 모드 스위치를 고정해, 버튼 문구가 상황별로 바뀌던 이전 구조보다 진입 목적이 명확해졌다.

### 3.2 로그인 맥락 분기

`src/pages/Login.js`의 `getLoginContext`는 접근하려던 경로에 따라 로그인 화면의 제목과 설명을 바꾼다.

- 관리자 영역 접근 시: "운영자 로그인"
- 비즈니스 영역 접근 시: "비즈니스 로그인"
- 일반 접근 시: "로그인"

이는 사용자가 왜 로그인 화면으로 이동했는지 이해하게 도와주는 좋은 패턴이다.

### 3.3 입점 신청 단계 구성

`src/pages/BusinessSignup.js`는 입점 신청을 단계형으로 제공한다.

- 계정
- 담당자
- 사업자
- 매장
- 메뉴
- 검토

중복 확인, 사업자등록번호 확인, 세션 임시 저장, 제출 전 확인까지 포함되어 있어 신청 과정 자체는 꽤 친절하다.

### 3.4 준비되지 않은 기능의 정직한 처리

서비스 의견과 콘텐츠 검증은 실제 기능이 연결되지 않았음을 명확히 말한다.

- `src/pages/Feedback.js`
- `src/pages/ContentVerification.js`

"가짜 통계나 동작하지 않는 제출 버튼은 노출하지 않습니다" 같은 방향은 신뢰를 만든다. 실제 서비스에서 중요한 태도다.

### 3.5 관리자 승인 흐름의 운영 맥락

`src/admin/pages/AdminStoreApprovals.js`는 필터, 상세 드로어, 권한별 액션 제한, 사업자 인증 완료 전 승인 제한, 동시 처리 충돌 메시지를 갖추고 있다.

운영자가 어떤 조건에서 승인할 수 있고, 왜 버튼이 막혔는지 알려주는 점이 좋다.

### 3.6 세션 갱신과 만료 안내

`src/api/client.js`와 `src/auth/AuthContext.js`는 토큰 갱신 이후 React 상태와 저장소를 동기화한다. 세션 만료 시 로그인 화면으로 이동하며 "로그인 세션이 만료되었습니다. 다시 로그인해 주세요."를 보여준다.

이전 기능 리뷰에서 지적될 수 있는 세션 갱신 리스크는 현재 코드 기준으로 상당 부분 해소되어 있다.

## 4. 주요 개선 필요 지점

### 4.1 입점 신청 이후의 맥락 부족

문제:

입점 신청 단계 자체는 친절하고, 신청 상세에는 상태별 다음 행동 안내와 미니 타임라인이 추가되었다. 다만 현재 안내는 프론트에서 계산한 상태 맥락 중심이어서, 서버가 기록한 실제 처리 이력과 보완 항목별 수정 흐름까지는 아직 부족하다.

- 실제 검토 이벤트는 언제 발생했는가?
- 보완 요청을 받으면 어느 항목을 어디에서 고쳐야 하는가?
- 반려 사유는 무엇이고 재신청할 수 있는가?
- 제출 버전 간 어떤 정보가 달라졌는가?
- 운영팀 코멘트와 신청자에게 보이는 안내가 어떻게 연결되는가?

관련 파일:

- `src/pages/BusinessApplications.js`
- `src/pages/BusinessApplicationDetail.js`

개선 방향:

- 서버 처리 이력 기반 타임라인 추가
- 보완 요청 사유와 수정 위치 연결
- 반려 후 재신청 안내 강화
- 제출 이력과 버전 이력 표시
- 관리자 보완/반려 메시지의 신청자 노출 방식 미리보기

권장 문구 예시:

- "운영팀이 사업자 정보와 매장 정보를 검토 중입니다. 보통 1-3영업일이 걸립니다."
- "보완 요청된 항목이 있습니다. 담당자 연락처와 매장 주소를 수정한 뒤 다시 제출해 주세요."
- "승인이 완료되었습니다. 이제 매장 기본 정보와 대표 메뉴를 관리할 수 있습니다."

### 4.2 Q&A의 공개 범위와 개인정보 맥락 부족

문제:

Q&A는 공개 목록과 질문 등록 화면이 분리되었고, 공개 범위와 이메일 사용 목적도 안내한다. 공개 목록은 게시판형으로 정리되어 검색과 비교가 쉬워졌으며, 1:1 비공개 문의 접수 화면도 별도 경로로 추가되었다. 다만 내가 남긴 비공개 문의를 다시 확인하거나, 답변 발송 여부와 처리 상태를 추적하는 기능은 아직 부족하다.

- 내가 남긴 문의만 모아볼 수 있는가?
- 답변 알림은 실제 이메일로 발송되는가?
- 공개 Q&A에 잘못 남긴 개인정보를 어떻게 보호할 것인가?

관련 파일:

- `src/pages/QnA.js`

개선 방향:

- 내 문의 목록과 비공개 문의 처리 상태 조회
- 로그인 사용자에게 "내 문의" 탭 제공
- 답변 예상 시간 안내
- 답변 이메일 알림 정책 명시
- 개인정보 포함 문의 경고와 운영자 비공개 전환 흐름 제공

권장 구조:

- 공개 Q&A: 다른 사용자도 볼 수 있는 일반 질문
- 1:1 문의: 계정, 결제, 개인정보, 사업자 정보 등 비공개 문의

### 4.3 용어 혼재

문제:

초기 검토 당시 서비스 안에서 같은 대상을 다음처럼 다르게 부르는 문제가 있었다.

- 식당
- 매장
- 가게

현재 상태:

- 주요 파트너/운영 화면은 `매장` 중심으로 정리되었다.
- 일반 사용자 맥락에서는 `식당`, 식당 파트너/운영자 맥락에서는 `매장`을 쓰는 방향이 대체로 맞춰졌다.
- `가게` 표현은 주요 화면에서 제거되었다.

관련 파일:

- `src/components/AppShell.js`
- `src/config/routes.js`
- `src/pages/RestaurantManagement.js`
- `src/pages/RestaurantDetail.js`
- `src/admin/pages/AdminStoreApprovals.js`

개선 방향:

권장:

- 일반 사용자 화면: "식당"
- 식당 파트너/운영자 화면: "매장"
- "가게"는 제거하거나 캐주얼 마케팅 영역에만 제한

완료된 통일 후보:

- "내 가게 관리" -> "내 매장 관리"
- "새 가게 등록" -> "새 입점 신청"
- "가게 상세 관리" -> "매장 상세 관리"

## 6. 화면별 개선 메모

### 로그인

좋은 점:

- 관리자/비즈니스/일반 로그인 맥락 분기
- 아이디 기억하기 제공
- 회원가입, 입점 신청, 문의하기 링크 제공
- 로그인 이후 일반 사용자와 식당 점주 진입을 고정된 모드 스위치로 제공

개선:

- 계정 문의 경로를 더 명확히 제공
- 관리자 로그인 실패 시 권한 부족과 비밀번호 오류를 구분할 수 있으면 좋음

### 일반 회원가입

좋은 점:

- 필수 동의와 기본 검증 제공

개선:

- 약관과 개인정보 처리방침 링크를 체크박스 옆에서 바로 열 수 있게 제공
- 비밀번호 조건을 입력 전부터 명시

### 식당 입점 신청

좋은 점:

- 단계형 신청
- 중복 확인
- 사업자 정보 확인
- 임시 저장
- 제출 전 검토

개선:

- 각 단계가 왜 필요한지 짧은 설명 추가
- 사업자 검증 실패 시 가능한 원인 안내
- 최종 제출 전 처리 예상 시간 안내
- 보완 요청을 받은 신청을 수정할 수 있는 흐름 강화

### 신청 현황과 상세

좋은 점:

- 심사 상태와 검증 상태를 분리해서 보여줌
- 승인 후 매장 관리 이동 링크 제공
- 상태별 다음 행동 안내 패널 제공
- 현재 진행 단계를 보여주는 미니 타임라인 제공
- `regionCode`, `categoryCode` 라벨 변환 반영

개선:

- 서버 처리 이력 기반 타임라인 추가
- 보완 사유와 항목별 수정 바로가기 추가
- 반려 후 다음 선택지 제공
- 제출 버전 비교

### 매장 관리

좋은 점:

- 검색, 카테고리, 노출 상태 필터 제공
- 메뉴 수, 수정일, 상태 표시

개선:

- "가게"와 "매장" 용어 통일
- 삭제를 공통 확인 다이얼로그로 변경
- 삭제 대신 비공개/보관 옵션 검토
- 노출 상태의 의미 안내는 반영했다. 삭제 대신 비공개/보관 옵션은 별도 검토가 필요하다.

### FAQ

좋은 점:

- 일반 사용자와 관리자 모드를 한 컴포넌트로 처리
- 관리자 등록/수정/삭제 제공
- 일반 사용자 화면을 게시판/아코디언형 목록으로 정리
- 분류/검색어 필터와 더 보기 동선 제공

개선:

- 콘솔 로그 제거, 삭제 다이얼로그, 오류 문구, FAQ 검색과 카테고리 필터는 반영했다.
- 다음 단계는 FAQ 카테고리 체계와 인기 FAQ/추천 FAQ 같은 탐색 보조 기능 검토다.

### Q&A

좋은 점:

- 비로그인 문의 접수 가능
- 관리자 답변 작성 가능
- 오류 시 다시 시도 제공
- 공개 Q&A 목록과 질문 등록 화면 분리
- 공개 Q&A를 게시판형 목록으로 표시
- 1:1 비공개 문의 접수 화면 제공
- 공개 범위, 개인정보 입력 주의, 이메일 사용 목적 안내 제공

개선:

- 내 문의 목록
- 답변 예상 시간 안내
- 답변 알림 정책 명시

### 공통 헤더와 고객지원 레이아웃

좋은 점:

- 헤더와 본문 최대 폭을 공통 기준으로 맞췄다.
- 헤더 배경과 본문 카드의 표면감, 그림자, 간격을 같은 톤으로 조정했다.
- 고객지원 화면에서 목록, 필터, 등록 CTA의 위치가 더 일관되게 정리되었다.

개선:

- 관리자 영역과 비즈니스 상세 화면까지 같은 디자인 언어가 유지되는지 추가 시각 점검이 필요하다.
- 모바일에서 긴 사용자 이름, 긴 질문 제목, 필터 조합이 레이아웃을 흔들지 않는지 실제 기기 기준 확인이 필요하다.

### 관리자 대시보드

좋은 점:

- KPI, 추이, 지역 분포, 최근 활동 구성
- 승인 대기 확인 CTA 제공

개선:

- 전체 활동 보기 기능 연결
- 오늘 처리해야 할 작업 큐 추가
- 알림 센터 연결
- 데이터 집계 기준 기간을 화면에서 더 명확히 표시

### 승인 관리

좋은 점:

- 필터, 상세 드로어, 승인/보류/반려 처리 흐름이 좋음
- 사업자 인증 완료 전 승인 제한
- 조회 권한 사용자에게 액션 제한 사유 표시

개선:

- 보류/반려 메시지가 신청자에게 어떻게 보이는지 미리보기
- 처리 이력 타임라인
- 서류 미리보기 실패 시 안내
- 대량 선택 체크박스의 실제 기능 연결 또는 숨김

### 회원 모니터링

좋은 점:

- 운영 체크포인트가 명확함
- 위험 계정, 로그인 이상 징후, 권한 변경 이력을 구분

개선:

- 위험 계정 상세 보기
- 조치 기록 남기기
- 계정 잠금, 권한 회수, 메모 기능
- 운영자별 처리 상태

4. 개발자 중심 오류 문구 수정 - FAQ/Q&A 중심 반영
5. 실제 동작하지 않는 버튼의 준비 중 처리 - 관리자 알림/전체 활동 보기 반영
6. 일반 사용자/식당 점주 진입 모드 정리 - 반영

### 2단계: 상태 맥락 강화

1. 입점 신청 서버 처리 이력과 제출 버전 비교
2. 보완 요청 상세와 재제출 흐름
3. 승인 후 매장 관리 안내
4. 반려 후 재신청 안내
5. Q&A 답변 예상 시간 안내와 답변 알림 정책

### 3단계: 운영 효율 기능

1. 관리자 알림 센터
2. 오늘 할 일 작업 큐
3. 내 문의 목록과 비공개 문의 처리 상태 조회
4. 미디어 관리 고도화
5. 회원 위험 계정 조치 기능

## 9. 최종 판단

현재 Plate Service는 기능적으로 기본 골격이 잘 잡혀 있고, 사용자를 속이지 않으려는 정직한 UI 방향도 좋다. 최근 개선으로 고객지원 목록/등록 동선, 일반 사용자/식당 점주 모드 전환, 헤더와 본문 디자인 정합성도 좋아졌다. 특히 입점 신청과 관리자 승인 관리의 구조는 실제 서비스 운영으로 이어질 수 있는 수준이다.

다만 사용자가 "지금 무엇을 기다리고 있는지", "다음에 무엇을 해야 하는지", "내 정보가 어디까지 공개되는지"를 이해하기 위한 기본 안내는 개선되었지만, 서버 처리 이력, 내 문의 목록, 답변 알림, 보완 항목별 수정 같은 실제 운영 기능은 아직 남아 있다.

따라서 다음 개선의 중심은 새 화면을 무작정 늘리는 것보다, 기존 화면의 상태 데이터를 서버 이력과 연결하고 실제 운영 행동으로 이어지게 만드는 것이다. 다음 우선순위는 보완 요청 흐름, 내 문의 목록과 답변 알림, 관리자 알림 센터/작업 큐, 미디어 관리 고도화로 잡는 것을 권장한다.
