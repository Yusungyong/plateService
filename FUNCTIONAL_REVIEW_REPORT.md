# Plate Service 전체 기능 검토 보고서

- 작성일: 2026-06-22
- 검토 대상: `C:\workspace\plateService` 프런트엔드
- 검토 방식: 소스 정적 검토, 자동 테스트, 프로덕션 빌드, 운영 서버 비인증 스모크 테스트
- 종합 판정: **조건부 적합**

## 1. 요약

Plate Service는 일반 사용자, 입점 신청자, 매장 운영자, 내부 관리자를 구분하는 라우팅과 권한 구조를 갖추고 있다. 로그인, 일반 회원가입, 사업자 입점 신청, 신청 현황, 매장 관리, FAQ/Q&A, 관리자 대시보드와 승인 관리는 실제 API 호출 구조까지 구현되어 있다.

다만 운영 배포 전에 다음 사항을 우선 해결해야 한다.

1. 액세스 토큰 갱신 결과가 React 인증 상태와 로컬 저장소에 동기화되지 않는다.
2. 인증 계정을 사용한 서버 연동 및 쓰기 E2E 검사가 수행되지 않았다.
3. 서비스 의견, 콘텐츠 검증, 관리자 매장·피드·시즌 큐레이션은 시연 또는 자리표시자 단계다.
4. Q&A의 API 실패 시 예시 데이터가 표시되어 실제 장애를 정상 데이터처럼 보이게 할 수 있다.
5. 자동 테스트 커버리지가 약 42%로 주요 운영 기능의 회귀를 충분히 방어하지 못한다.

현재 상태는 UI 검토 및 스테이징 연동에는 적합하지만, 전체 기능을 운영 환경에 일반 공개하기 전에는 우선순위 P1 항목의 조치가 필요하다.

## 4. 주요 발견 사항

### P1 — 운영 전 필수

#### 4.1 토큰 갱신 상태 동기화 누락

`src/api/client.js`는 액세스 토큰 갱신 후 모듈 내부 토큰만 교체한다. `src/auth/AuthContext.js`의 `authState`와 `localStorage`에는 새 토큰이 반영되지 않는다.

영향:

- 현재 탭에서는 갱신 토큰으로 요청할 수 있지만 React가 보유한 사용자 정보는 오래된 상태로 남을 수 있다.
- 새로고침하면 만료된 액세스 토큰과 이전 리프레시 토큰이 다시 복원된다.
- 서버가 리프레시 토큰을 회전시키는 경우 새로고침 후 세션 복구가 실패할 가능성이 높다.

권고:

- API 클라이언트에 `onAuthSessionRefreshed` 콜백을 추가한다.
- 갱신 성공 시 AuthContext 상태와 저장소를 원자적으로 업데이트한다.
- 토큰 회전, 동시 401, 갱신 실패, 새로고침 복원 테스트를 추가한다.

#### 4.2 인증 기반 운영 서버 E2E 검증 공백

운영 서버 스모크 테스트는 비인증 차단만 확인했다. 로그인, 관리자 조회, 사업자 매장 목록, 업로드, 생성·수정·삭제는 계정 정보가 없어 건너뛰었다.

권고:

- 읽기 전용 관리자/사업자 테스트 계정을 별도로 준비한다.
- 쓰기 테스트는 격리된 테스트 매장을 생성한 뒤 반드시 삭제하도록 유지한다.
- 현재 UI가 사용하는 `/api/owner/stores`와 입점 신청 API를 스모크 대상에 추가한다.

#### 4.3 미완성 메뉴의 운영 노출

관리자 매장 관리, 피드 관리, 시즌 큐레이션이 내비게이션에 노출되지만 실제 기능은 자리표시자다. 서비스 의견과 콘텐츠 검증도 사용자가 동작 가능한 화면처럼 보이나 실제 저장 기능이 없다.

권고:

- 구현 전까지 메뉴를 기능 플래그로 숨기거나 명확히 `준비 중`으로 표시한다.
- 동작하지 않는 버튼은 제거 또는 비활성 처리하고 사유를 제공한다.

### P2 — 단기 개선

#### 4.4 Q&A 장애 은폐 가능성

Q&A 목록 API가 실패하면 고정 예시 데이터가 대신 표시된다. 사용자는 서버 장애를 정상 조회 결과로 오인할 수 있다.

권고:

- 운영 환경에서는 예시 데이터 대신 오류와 재시도 버튼을 표시한다.
- Mock 데이터는 테스트 또는 Storybook 성격의 환경으로만 제한한다.

#### 4.5 권한 모델의 세분화 불일치

`FAQ_MANAGE`, `QNA_MANAGE`, `MEMBER_MONITORING_READ` 권한이 정의되어 있지만 실제 관리자 라우트는 공통 `SUPPORT_MANAGE`를 사용한다.

영향:

- 서버의 세분화된 권한 정책과 프런트 메뉴 노출이 달라질 수 있다.

권고:

- 각 화면을 전용 권한에 연결하거나 `SUPPORT_MANAGE`가 상위 권한이라는 계약을 명문화한다.

#### 4.6 API 오류 처리 일관성 부족

- FAQ 상세 조회 실패는 사용자 메시지 없이 상세만 비운다.
- 리프레시 토큰이 없거나 일부 401 코드가 허용 목록에 없으면 전역 로그아웃 처리가 실행되지 않는다.
- 전역 Error Boundary가 없어 렌더링 예외 시 전체 화면이 중단될 수 있다.

권고:

- 공통 오류 화면, 재시도, 인증 만료 처리 규칙을 통합한다.
- 최상위 및 주요 라우트 단위 Error Boundary를 추가한다.

#### 4.7 자동 테스트 범위 부족

전체 테스트는 통과하지만 라인 커버리지는 42.62%, 분기 커버리지는 32.31%다.

현재 강점:

- 관리자 권한과 승인 처리
- 사업자 공개 가입과 계정 중복 확인
- 오래된 비동기 응답 무시
- 사업자/관리자 라우트 격리
- JWT 한글 표시명 처리

부족한 범위:

- 토큰 갱신과 만료
- FAQ/Q&A CRUD 및 오류 상태
- 회원 모니터링
- 매장 상세 수정·삭제·미디어 업로드
- 입점 신청 보완 제출과 충돌 응답
- 반응형 화면과 접근성 회귀

#### 4.8 운영 로그 정리 필요

FAQ 생성 시 요청 payload와 응답을 `console.log`로 출력한다. 운영 환경에서 불필요한 데이터가 브라우저 콘솔에 남을 수 있다.

권고:

- 해당 로그를 제거하거나 개발 환경 전용 로거로 제한한다.

### P3 — 유지보수

#### 4.9 도구 체인 경고

- React Router v7 전환 관련 Future Flag 경고가 테스트마다 발생한다.
- Create React App의 Babel preset이 선언되지 않은 플러그인에 의존한다는 경고가 발생한다.
- Browserslist 데이터가 오래되었다는 빌드 경고가 발생한다.

권고:

- 단기적으로 React Router Future Flag를 검토한다.
- Browserslist 데이터를 갱신한다.
- 중기적으로 유지보수가 중단된 CRA에서 Vite 등 현재 유지되는 빌드 체계로 이전을 계획한다.

#### 4.10 브라우저 저장소의 토큰 보관

액세스 토큰과 리프레시 토큰을 `localStorage`에 보관한다. XSS가 발생하면 토큰 탈취 위험이 있다.

권고:

- 서버 구조가 허용하면 리프레시 토큰을 HttpOnly/Secure/SameSite 쿠키로 이전한다.
- CSP와 입력 데이터 출력 정책을 함께 강화한다.

## 5. 설계상 강점

- 공개, 인증 사용자, 사업자, 관리자를 라우트 수준에서 명확히 분리했다.
- 관리자 권한은 단순 역할뿐 아니라 세부 permission까지 검사한다.
- API 호출을 도메인별 모듈로 분리했고 응답 정규화 계층이 존재한다.
- 사업자 가입 임시 저장 시 비밀번호를 제외한다.
- 사업자 계정 중복 검사에서 오래된 비동기 응답을 무시하도록 방어했다.
- 목록, 상세, 로딩, 빈 상태, 오류 상태가 주요 화면에 마련되어 있다.
- 관리자 Mock은 환경변수와 테스트 환경으로 제한되어 운영 API와 분리된다.
- 반응형 레이아웃, 폼 레이블, 상태 메시지 등 기본 접근성 구조가 대체로 갖춰져 있다.

## 6. 권장 실행 순서

### 1단계: 운영 차단 위험 제거

1. 토큰 갱신 결과를 AuthContext와 저장소에 동기화한다.
2. 운영 서버 CORS에 실제 프런트 Origin을 등록하고 재검증한다.
3. 사업자·관리자 계정으로 읽기 E2E를 수행한다.
4. 격리 데이터로 입점 신청 및 매장 CRUD 쓰기 E2E를 수행한다.

### 2단계: 사용자 오인 방지

1. 미구현 관리자 메뉴를 숨기거나 준비 중 표시를 강화한다.
2. 서비스 의견과 콘텐츠 검증의 동작하지 않는 버튼을 정리한다.
3. Q&A의 운영 fallback 데이터를 제거한다.
4. FAQ 콘솔 로그와 무음 오류를 정리한다.

### 3단계: 회귀 방어 강화

1. 인증 갱신, FAQ/Q&A, 매장 상세, 모니터링 테스트를 우선 추가한다.
2. 핵심 도메인 라인 커버리지 70% 이상을 1차 목표로 설정한다.
3. 실제 브라우저 E2E와 접근성 검사를 CI에 추가한다.
4. CRA 및 React Router 전환 경고를 단계적으로 해소한다.

## 7. 최종 판정

핵심 기능의 구조와 주요 사용자 흐름은 잘 갖춰져 있으며 자동 테스트와 빌드도 안정적으로 통과한다. 특히 사업자 입점 신청과 관리자 승인 관리의 구현 완성도는 비교적 높다.

그러나 인증 토큰 갱신 동기화 문제, 인증 기반 서버 검증 부재, 운영 메뉴에 노출된 미완성 기능 때문에 현재 상태를 전체 운영 기능 완료로 판정하기는 어렵다.

**판정: 조건부 적합 — P1 항목 해결 및 인증 E2E 통과 후 운영 승인 권장**
